Strict Session UI Study

Allow List stays default. Blocked List becomes the new second mode.

This page turns the earlier product discussion into concrete screens. It shows how the options page can hold both list types, how the blocked list can stay hidden by default behind a top-right eye control, and how the popup can expose different session-mode toggles without making the extension feel heavier.

Allow List remains the default path Blocked List is a real session mode Settings privacy toggle hides blocked entries by default Popup mode switch is the fastest recommended control

Recommended direction

The cleanest product shape is to keep the existing allowlist mental model intact, layer blocklist on top as a second mode, and avoid mixing both lists into one overloaded screen.

Settings layout Use top tabs: Allow List first, Blocked List second. It maps well to the current single-list page.
Blocked list privacy Keep blocked entries blurred and dimmed on page load. Reveal them only after clicking the top-right eye button.
Session type control Put an Allow / Blocked segmented control in the popup above the duration grid so the choice is obvious at start time.
Runtime behavior Allow mode keeps the current redirect-all-except-allow behavior. Blocked mode redirects only domains found in Blocked List.

Behavior and Storage Decisions

These are the underlying product rules the UI is expressing. Writing them down keeps the design aligned with the extension logic.

Topic Decision Why it fits this extension
Default mode Keep Allow List as the default selected tab, default popup mode, and default stored preference. It preserves current behavior and avoids surprising existing users.
New data Add a second array for blockedList plus a stored mode value such as sessionMode. The current extension already stores allowedList, so this is the smallest clean extension of the model.
Allow mode logic Keep the current rule system: redirect everything during a session, then add allow rules for each allowed domain. No migration risk for existing users.
Blocked mode logic Reverse the matching strategy: only redirect main-frame requests that match entries in blockedList. Technically simple with the same declarative net request approach already used in the background worker.
Blocked list privacy Hide blocked entries by default in settings with a local eye-toggle state that resets when the page reopens. Users get privacy without creating a persistent hidden-state setting.
Popup recommendation Use a segmented Allow / Blocked selector directly above the duration buttons. It adds one fast decision point without changing the flow of starting a session.

Implementation Notes To Pair With The Designs

These notes bridge the visuals back to the repo so the next step from mockup to code is straightforward.

Options Page

  • Reuse the current input and list rendering structure, but parameterize it by active tab.
  • Add a small tab state layer so the page can swap titles, helper copy, empty states, and active list.
  • Apply the eye toggle only to Blocked List content and default it to hidden on every page load.

Popup

  • Show the current mode prominently before users choose a duration.
  • If segmented toggle is used, keep Allow active by default and mirror the selection into session start messages.
  • Small helper text under the toggle can explain the selected mode in one sentence.

Background Logic

  • Carry both lists in storage, normalize both lists the same way, and decide which rule set to apply from sessionMode.
  • Allow mode remains catch-all redirect plus allow exceptions.
  • Blocked mode becomes domain-specific redirect rules with no global redirect rule.
